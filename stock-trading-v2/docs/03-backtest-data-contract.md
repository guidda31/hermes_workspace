# 자동매매 V2 — 백테스트 데이터 계약 v0

> 목적: 전략·리스크 규칙을 코드로 구현하기 전에, 백테스트가 필요한 입력 데이터와 시간 정합성 기준을 고정한다.

## 1. 일별 가격 데이터

각 자산·거래일별 필수 컬럼:

| 컬럼 | 설명 |
|---|---|
| `trade_date` | KST 거래일 |
| `symbol` | 표준 종목 코드 |
| `asset_type` | `STOCK` 또는 `ETF` |
| `open`, `high`, `low`, `close` | 조정 전 원시 일봉 가격 |
| `volume` | 거래량 |
| `trading_value` | 거래대금 |
| `is_tradable` | 해당 일 주문 가능 여부 |
| `data_timestamp` | 데이터 수집·확정 시각 |

신호는 `t`일 장 마감 확정 데이터만 사용하고, 체결은 `t+1` 거래일에만 가정한다. `t`일 종가로 신호를 낸 뒤 같은 `t`일 종가로 체결하는 방식은 금지한다.

## 2. 종목 마스터·상태 데이터

- 종목 코드·명·시장(KOSPI/KOSDAQ)
- 자산 유형(보통주/ETF/ETN 등)
- 상장일·상장폐지일·코드 변경 이력
- ETF 분류(일반 지수형/섹터형/레버리지/인버스/해외지수 추종)
- 우선주·스팩·관리·거래정지·주문 제한 상태
- 섹터 분류

이 데이터는 각 거래일 기준으로 재현 가능해야 한다. 현재의 종목 상태를 과거에 소급 적용하지 않는다.

## 3. 시장 기준 데이터

- KOSPI 200 일봉
- KOSPI·KOSDAQ 일봉
- 거래일 달력
- 거래정지·가격제한폭·기업행사 이벤트

시장 필터의 기준 지수는 백테스트 시작 전 하나로 고정하고, 이후 검증 구간에서 임의 변경하지 않는다.

## 4. 체결·비용 가정

최소 두 가지 시나리오를 병행한다.

1. **다음 거래일 시가 + 보수적 슬리피지**
2. **지정가 체결 + 미체결 처리**

공통 반영 요소:
- 주식·ETF별 수수료, 세금, 기타 거래비용
- 호가 단위와 가격제한폭
- 거래대금 참여율 한도
- 갭상승 매수 보류 규칙
- 거래정지·주문 불가·미체결·부분 체결 처리

## 5. 데이터 품질 게이트

아래 조건을 만족하지 않으면 해당 자산·거래일을 신호 대상에서 제외하고 사유를 기록한다.

- 필수 OHLCV·거래대금이 존재
- 가격의 비정상값(0 이하, 고가/저가 관계 오류 등)이 없음
- 상장 전·상장폐지 후 데이터가 아님
- 거래 가능 상태가 확인됨
- 분할·배당 등 가격 조정 기준이 문서화됨

## 6. 백테스트 출력 계약

각 신호·주문·체결·포지션·청산 건마다 다음을 보존한다.

- `strategy_id`, `signal_id`, `order_id`, `position_id`
- 신호 시점의 모든 입력 지표와 후보 순위
- 주문 가정 가격·수량·미체결 여부·슬리피지·비용
- 진입/청산 사유
- 실현 손익·평가 손익·보유 기간
- 시장 필터 상태와 리스크 차단 여부

## 7. 데이터 공급원 결정

- 1차 백테스트 데이터: **공개 KRX 일봉 데이터**
- 초기 구현: `FinanceDataReader`의 공개 OHLCV 조회 어댑터
- 주의: 이 공급원은 거래대금 필드를 제공하지 않으므로, 1차 유동성 필터에서는 `종가 × 거래량`을 **추정 거래대금 프록시**로 사용한다. 이는 거래소 공식 거래대금이 아니며, 페이퍼 트레이딩 전에는 공식 거래대금 데이터로 교체·대사한다.
- 목적: 전략 가설 검증을 주문 API·실거래와 분리해 수행한다.
- 이후 페이퍼 트레이딩과 실주문 단계에서만 브로커 시세·주문 API를 별도 어댑터로 연결한다.

## 8. 다음 구현 게이트

백테스트 코드는 데이터 계약의 필수 컬럼·품질 게이트·다음 거래일 체결 원칙을 자동 검증해야 한다. 데이터 공급원은 이 계약을 충족하는지 확인한 뒤 선택한다.
