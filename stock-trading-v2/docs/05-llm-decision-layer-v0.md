# 자동매매 V2 — LLM 판단 계층 v0 설계

> 상태: **설계 가설(H)**. 실주문·성과주장·자동화 없음. **Hermes 에이전트(OpenAI OAuth 경유 GPT)** 가 시점정합(PIT) 데이터 브리프를 해석해 매매를 **제안**하고, 하드 리스크 가드레일이 이를 강제하며, v0는 **주문 없는 forward observation**으로만 검증한다.
>
> 이 문서는 `00~04` 설계 원칙을 계승한다. 특히 "실주문은 백테스트·페이퍼 검증 통과 전까지 비활성", "모든 판단은 재현 가능한 로그", "다음 거래일 체결 원칙", "deny-by-default 유니버스"를 그대로 따른다.

## 0. 아키텍처 전제 — Hermes가 곧 LLM (별도 API 연결 없음)

**확정:** 본 프로젝트는 Hermes 기반으로 동작하며, Hermes는 이미 **OpenAI OAuth로 GPT에 인증**되어 있다(Codex 런타임, `openai/gpt-5.5`). 따라서 매매의 **LLM 판단 역할은 Hermes 에이전트 자신**이 수행한다.

- **우리 저장소 코드는 LLM API를 호출하지 않는다.** `openai`/OpenRouter SDK·API 키를 두지 않는다. 판단 지능은 Hermes 런타임에 있다.
- 역할이 반전된다: **코드가 GPT를 호출하는 게 아니라, GPT(=Hermes)가 코드를 호출**한다.
- 저장소가 제공하는 것은 Hermes가 호출하는 **결정론적 도구**들이다: PIT 데이터 브리프 생성, 결정 가드레일 검증, 신호 감사 기록, (게이트 뒤) 주문 제출.

| 구성 | 역할 |
|---|---|
| Hermes 에이전트 (GPT via OAuth) | 판단하는 두뇌 — 브리프를 읽고 종목 선별·매매 결정 |
| 저장소(파이썬) | Hermes가 호출하는 결정론적 도구: 브리프·가드레일·감사·(게이트)주문 |

기존 코드의 매매 판단은 순수 규칙(`signals.py`의 SMA50/200·돌파)이다. 이를 **Hermes의 판단으로 대체/보강**하되 실행·리스크·감사 인프라는 재사용한다.

재사용(그대로): KIS 데이터 수집, 리스크 가드(5종목·1%·3%), 감사/로그, no-lookahead 시간모델, (비활성) 주문 제출 배관.
대체: `signals.py` 규칙 → Hermes 판단.
신규: 공시·뉴스 수집, PIT 브리프 빌더, 결정 스키마·가드레일 검증 도구, 신호 감사, forward 러너.

## 1. 최우선 위험 — LLM 특유의 검증 무효화 (Lookahead / Hindsight)

**확정 원칙:** LLM은 학습 데이터에 미래 지식이 포함될 수 있으므로, 과거 구간 백테스트는 **오염 가능**하며 그 자체로 실거래 기대값의 증거가 아니다.

이를 통제하기 위해 두 층위로 방어한다.

1. **데이터 레벨 PIT 차단(주 방어):** 신호일 `t`의 판단에는 `t`까지 확정·발행된 정보만 컨텍스트에 넣는다. 가격은 `t` 종가까지, 공시·뉴스는 **발행시각 ≤ `t` 장 마감**인 항목만. 발행시각이 증명되지 않는 항목은 컨텍스트에서 제외한다. 프롬프트로 "미래를 모른 척하라"고 지시하는 것만으로는 **불충분**하다.
2. **검증 방법론(정직한 신뢰):** v0의 1차 신뢰 근거는 과거 백테스트가 아니라 **forward observation**(실제 미래 거래일을, 결과를 모른 채, 주문 없이 신호만 기록하고 다음날 실제 결과와 대사)이다. 과거 백테스트는 오직 (a) 파이프라인 정합성 점검과 (b) 규칙 baseline 대비 참고용으로만 쓰고, 오염 가능성을 리포트에 명시한다.

**확정 원칙:** 신호일 이후 발행된 어떤 공시·뉴스·가격도 그 신호일 판단 컨텍스트·근거·순위에 들어갈 수 없다. 위반 시 해당 신호는 무효로 기록한다.

## 2. 아키텍처 (데이터 → 판단 → 가드레일 → 감사)

```text
[가격/거래대금]  [DART 공시]  [뉴스]
      \             |            /
       └──> PIT 브리프 빌더 (신호일 t까지, 발행시각 검증) ──┐   ← 저장소 도구
                                                            v
                                Hermes 에이전트 (GPT via OAuth) ← 두뇌, API 호출 아님
                                                            │  구조화 결정(도구 경계에서 스키마 검증)
                                                            v
              하드 리스크 가드레일 (기존 5종목·1%·3%·유동성·유니버스 화이트리스트) ← 저장소 도구
                                                            │
                                                            v
              신호 감사 기록 (immutable, 주문/체결/포지션 필드 없음, 브리프·결정·모델·시각 보존) ← 저장소 도구
```

Hermes를 매 거래일 구동하는 트리거는 **Hermes 루틴/cron**이다(현재 비활성, 별도 승인 전 재활성 금지). 루틴이 브리프 도구 → (Hermes 판단) → 가드레일 도구 → 감사 도구 순으로 호출한다.

**확정:** `t` 종가 확정 후 판단 → `t+1` 시가 체결을 **가정한 신호만** 생성(실주문 아님). 기존 시간모델과 동일.

## 3. 데이터 소스 계약

각 항목은 반드시 **발행/확정 시각**과 **출처**를 가진다. 없으면 PIT 불가로 제외.

| 소스 | 상태 | PIT 근거 | 비고 |
|---|---|---|---|
| 가격·거래대금 | ✅ 있음 (KIS 일봉 / FDR) | `trade_date` 종가 확정 | `trading_value=close×volume` 프록시임 명시 |
| DART 전자공시 | 🆕 신규 | 접수시각 `rcept_dt`/`rcept_no` | DART OpenAPI(공식·무료). API 키는 KIS처럼 **외부 주입, 저장소·로그 금지** |
| 뉴스 | 🆕 신규(최난이도) | 기사 발행 타임스탬프 필수 | 발행시각 PIT 보장 가능한 소스만 채택. 검색시점≠발행시점 주의. 소스 미확정 시 인터페이스만 두고 빈 채로 시작 |

**확정:** 뉴스/공시 아이템은 `published_at`(UTC 또는 KST 명시)이 없으면 로드 거부. 과거 구간에 대해 "오늘 검색되니 그때도 있었다"고 가정 금지(KRX PIT 조사 결론과 동일 원칙, `krx-historical-metadata-source-recon.md` 참조).

## 4. 입출력 계약 (Hermes ↔ 저장소 도구)

### 4.1 브리프 (저장소 → Hermes, 신호일 t)
브리프 빌더 도구가 아래를 담은 PIT 브리프를 만들어 Hermes에 제공한다.
- `signal_date`, 대상 유니버스(기존 deny-by-default 통과 종목만)
- 종목별: 가격 요약(최근 N일 OHLCV·거래대금·이동평균·수익률), `t`까지 발행된 공시 목록(요약), `t`까지 발행된 뉴스 요약. 각 항목은 인용용 `evidence_id`를 가진다.
- 현재 포트폴리오(보유·평단·비중), 가용현금, 리스크 상태(일손실 차단 여부·잔여 슬롯)
- 시장 상태(지수 추세 등)

### 4.2 결정 (Hermes → 저장소 도구, 스키마 검증)
Hermes가 산출해야 하는 종목별 결정 객체 목록:
```text
{ symbol, action: BUY|SELL|HOLD, conviction: 0..1,
  target_weight: Decimal, rationale: str, cited_evidence: [evidence_id...] }
```
- **확정:** 결정은 반드시 이 구조화 스키마로만 받는다(자유 텍스트 매매지시 금지). 스키마 위반은 도구 경계에서 거부한다. Hermes가 API 호출이 아니라 에이전트로서 판단하므로, 스키마 강제는 **도구 입력 검증**으로 구현한다(structured-output API 파라미터가 아님).
- **확정:** `cited_evidence`는 브리프에 실제 제공된 `evidence_id`만 인용 가능. 미제공 사실 인용(환각)은 해당 결정 감점·기록.
- `rationale`+`cited_evidence`로 **모든 판단이 사후 감사 가능**해야 한다.

## 5. 하드 가드레일 (LLM 위의 코드 강제)

**확정:** LLM 출력은 **제안일 뿐**이며, 아래 한도는 코드가 강제한다. LLM이 위반을 제안하면 절사 또는 거부하고 사유를 기록한다.

- 유니버스 화이트리스트 밖 종목 제안 → 거부(ETN/레버리지/인버스/해외ETF/우선주/스팩/관리/정지/미분류 제외).
- 동시 5종목, 포지션당 위험 1%, 일손실 3% 신규진입 차단, 유동성 게이트 — 기존 `risk.py`/백테스트 리스크 재사용.
- 갭업 +5% 매수 보류(현재 미구현 → 신규 필요), 다음 거래일 가격 범위 초과 주문 금지.
- `target_weight` 합이 한도 초과 시 conviction 순 절사.

## 6. 재현성·감사

- **확정:** 실행마다 모델 ID(예: `openai/gpt-5.5`), 브리프 스냅샷 SHA-256, Hermes가 반환한 결정 원문, 시각(KST)을 기록한다.
- **에이전트 런타임 LLM은 API 호출처럼 고정·재실행할 수 없다.** `temperature`·seed로 핀할 수 없고 동일 재현이 불가하다. 따라서 재현성 대신 **감사 가능성**에 의존한다: 브리프·결정·모델·시각을 불변 보존해, 사후에 "무엇을 보고 무엇을 결정했는지"를 정확히 추적한다. 이 한계 때문에 §7의 forward observation 검증이 필수다.

## 7. 검증 게이트

1. **파이프라인 정합:** PIT 차단이 실제로 미래 아이템을 배제하는지 테스트로 증명.
2. **Forward observation:** 매 거래일 신호만 기록, 다음날 실제 결과와 대사. LLM 판단 품질(적중·근거타당성·환각율) 통계 축적.
3. **Baseline 대비:** 동일 기간 규칙기반(`signals.py`) 신호와 병행 비교.
4. N개월 forward 통과 + 독립 검토 후에만 paper. 실거래는 그 뒤 별도 명시 승인(기존 Phase 6 게이트).

## 8. 첫 구현 슬라이스 (TDD, 주문 없음, LLM API 없음)

전체 데이터 로더를 한 번에 만들지 않는다. **저장소는 도구만 만든다 — LLM API 클라이언트는 만들지 않는다.** 테스트는 Hermes 대신 fake 결정(dict)을 주입해 도구 경계를 검증한다.

1. **결정 스키마 + 검증 도구**: §4.2 결정 객체를 파싱·검증(스키마 위반·미제공 evidence 인용 거부). 입력은 dict(Hermes가 채울 자리), API 호출 없음.
2. **PIT 브리프 빌더**를 가격 데이터(기존 스냅샷)만으로 구현. 공시·뉴스는 인터페이스 stub(빈 목록). 미래 발행 항목 배제를 테스트로 증명.
3. **가드레일 도구** 연결: 기존 리스크로 결정 절사·거부. 유니버스 화이트리스트 밖 거부 테스트.
4. **신호 감사 기록**: 브리프·결정·모델·시각 보존, 주문/체결/포지션 필드가 0인지 테스트로 증명.
5. 이후 별도 RED→GREEN 주기로 DART 공시 수집 → 뉴스 수집 → Hermes 루틴 연동(forward 러너) 확장.

완료 기준: (a) 미래 발행 아이템이 브리프에 절대 안 들어감, (b) 결정이 스키마·화이트리스트·리스크 한도를 벗어나면 도구가 차단, (c) 모든 결정이 근거·인용·브리프 해시로 추적, (d) 저장소 어디에도 LLM/주문 API 호출 경로가 없음.

## 9. v0 비목표

실주문, 성과 기대값 주장, 오염되지 않았다는 과거 백테스트 수익률 주장, 뉴스/공시의 시점 미검증 사용, cron 자동화. 이들은 각각 별도 승인·검증 뒤에만 다룬다.
