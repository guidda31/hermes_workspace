기존 자동매매의 손실 원인을 전략 신호, 리스크 관리, 주문 실행, 시스템 장애로 분리해 정량적으로 확인한다. 이 단계에서는 신규 전략 개발·실계좌 주문을 수행하지 않는다.

## 현재 확인 결과
- 현재 작업 경로 `/home/guidda/hermes_workspace`에는 기존 KIS OpenAPI 소스와 거래 로그가 없다.
- 따라서 원본 거래·주문·체결·잔고·전략판단 로그를 확보한 뒤 분석을 시작한다.

## 확보할 원천 자료
1. 거래/체결 내역 CSV 또는 API 조회 결과
   - 주문 ID, 종목코드·명, 매수/매도, 주문·체결 시각, 주문·체결 가격, 주문·체결 수량, 수수료·세금, 주문 상태
2. 전략 판단 로그
   - 신호 시각, 진입/청산 근거, 당시 지표값, 후보군, 제외 사유, 전략 버전
3. 시세·호가 데이터
   - 신호 시각과 체결 시각의 현재가, 최우선 매수·매도 호가, 거래량·거래대금, 시장·업종 지수
4. 계좌/운영 이벤트 로그
   - 매매 전후 현금·보유수량, API 오류, 재시작, 주문 취소·정정, 중복 주문 방지 결과

## 분석 지표
- 수수료·세금 포함 순손익, 누적 손익, 기간 수익률
- 승률, 평균 수익, 평균 손실, 손익비, Profit Factor
- 최대낙폭(MDD), 일별 손실, 연속 손실 횟수
- 진입 시점·시간대·시장 국면별 손익
- 주문가 대비 체결가 차이(슬리피지), 미체결·부분체결·취소율
- 종목 유동성·스프레드·거래대금 구간별 손익

## 원인 분류
- `SIGNAL`: 시장/종목 선택 또는 진입·청산 신호의 기대값 문제
- `RISK`: 포지션 크기, 손절, 익절, 일 손실 중단 규칙 문제
- `EXECUTION`: 슬리피지, 호가 스프레드, 부분체결, 중복/지연 주문 문제
- `SYSTEM`: API, 네트워크, 재시작, 상태 동기화, 로그 누락 문제
- `DATA`: 지연·결측·잘못된 시세를 근거로 한 판단 문제

## 완료 기준
- 전체 거래를 주문 ID 기준으로 재구성한다.
- 모든 거래에 순손익·원인 분류·근거 데이터 존재 여부를 부여한다.
- 상위 손실 원인 3개와 각각의 금액/빈도 비중을 산출한다.
- 다음 단계의 신규 전략 요구사항으로 바꿀 수 있는 개선 항목을 최소 5개 도출한다.
